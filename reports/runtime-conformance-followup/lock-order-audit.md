# Runtime task transaction lock-order audit

Canonical row-lock order:

`provider_task -> task_command -> task_input_request -> task_input_response_inbox -> task_observation -> outbox_event -> provider_ops_delivery`

| Method                                             | Tables touched / row-lock order                                                                                                                 | Safe-stop interaction                                                                    | Result                                                                       |
| -------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| `acceptMcpInputResponses`                          | `provider_task -> task_input_request -> task_input_response_inbox -> task_observation -> outbox_event -> provider_ops_delivery`                 | Task lock serializes terminal and stop transitions before acceptance                     | Modified: locked Schema validation precedes Inbox insert                     |
| `beginCancel`                                      | `provider_task -> task_command -> task_observation -> outbox_event -> provider_ops_delivery`                                                    | Establishes the authoritative safe-stop state while Task is locked                       | Existing order retained                                                      |
| `beginCommand`                                     | `provider_task -> task_command -> task_observation -> outbox_event -> provider_ops_delivery`                                                    | Refuses normal commands after stop state is visible                                      | Existing order retained                                                      |
| `promotePendingInputResponses`                     | Unlocked candidate IDs; per Task: `provider_task -> task_input_response_inbox -> task_command insert -> outbox_event -> provider_ops_delivery`  | Rechecks terminal/stop state under Task lock; `SKIP LOCKED` prevents duplicate promotion | Modified to two phases and one transaction per sorted Task UUID              |
| `acknowledgeUpdateAndCompleteInputAnswers`         | `provider_task -> task_command -> task_input_request -> task_input_response_inbox -> task_observation -> outbox_event -> provider_ops_delivery` | Terminal/stop state exhausts Command and ignores assigned Inbox                          | Modified: Task lock now precedes Command lock                                |
| `rejectInputResponseAndFailTask`                   | `provider_task -> task_command -> task_input_request -> task_input_response_inbox -> task_observation -> outbox_event -> provider_ops_delivery` | Existing terminal or safe stop wins; no Task terminal state is overwritten               | New canonical transaction                                                    |
| `failInputResponseDelivery`                        | `provider_task -> task_command -> task_input_request -> task_input_response_inbox -> task_observation -> outbox_event -> provider_ops_delivery` | Safe stop changes delivery exhaustion into supersession and preserves STOPPING           | Modified for safe-stop priority and canonical order                          |
| `supersedeClaimedCommandForSafeStop`               | `provider_task -> task_command -> task_input_response_inbox -> outbox_event -> provider_ops_delivery`                                           | Runs only when safe stop is active                                                       | Existing order retained; no higher lock after Command                        |
| `supersedeExpiredClaimedNormalCommandsForSafeStop` | `provider_task -> task_command -> task_input_response_inbox -> outbox_event -> provider_ops_delivery`                                           | Expires normal commands after stop arbitration                                           | Existing order retained                                                      |
| `applySnapshot`                                    | `provider_task -> task_input_request -> task_observation -> outbox_event -> provider_ops_delivery`                                              | Terminal snapshots are serialized by Task lock                                           | Existing order retained                                                      |
| `claimDueCommands`                                 | `task_command` claim, then lower-tier Inbox/Outbox/Provider Ops writes                                                                          | Reads stop state for eligibility and supersedes expired normal claims                    | Allowed batch-claim exception; never obtains a later Task row lock           |
| `recordCommandStarted`                             | `task_command -> outbox_event -> provider_ops_delivery`                                                                                         | Claim ownership is verified                                                              | Allowed command-only transaction; Task is snapshot-read without `FOR UPDATE` |
| `retryClaimedCommand`                              | `task_command -> outbox_event -> provider_ops_delivery`                                                                                         | Dispatcher checks safe stop before retry                                                 | Allowed command-only transaction; Task is snapshot-read without `FOR UPDATE` |

`task_observation`, `outbox_event`, and `provider_ops_delivery` inserts are append-only in these
paths. Tests set `lock_timeout='3s'` and `statement_timeout='10s'`; they do not catch or suppress
PostgreSQL `40P01` errors.
