# Performance and capacity baseline

`pnpm capacity:baseline` creates an isolated PostgreSQL 17 schema and runs one Runtime with
`DATABASE_POOL_MAX=1`, one real TypeScript gRPC Adapter and one official MCP Streamable HTTP
client. It writes environment, topology, throughput and latency to
`reports/capacity/runtime-v1.json`, then drops the schema.

The rc.2 workload records and enforces:

- 100 sequential synchronous Tool calls and 25 durable create-plus-get lifecycles;
- SQL progress while a real Adapter delays StartOperation for 750 ms; the pool query must finish
  in under 500 ms and before the Adapter response;
- one dispatcher tick over 20 persisted cancel commands, all of which must be acknowledged;
- a 32-Task future scheduled scan and a 16-Task overdue immediate-start watchdog scan;
- real repository recovery candidate selection/materialization at 100, 500 and 1000 rows;
- Observation and Outbox row plus relation-size growth generated by the workload;
- the authoritative Linux image size, non-root user and reproducible production-only policy.

This is a regression baseline, not a production SLO. The recovery measurement covers PostgreSQL
candidate selection and Task materialization; real Adapter reconciliation latency is
resource-specific. It has no multi-tenant network contention or Outbox consumer. Load-test the
production Adapter and database before setting SLOs, and investigate any material regression on
the same CI runner/topology even if correctness thresholds still pass.

The authoritative image report is `reports/image/runtime-v1-rc2.json`: GitHub Actions Docker
Engine measured 327,026,557 bytes against a 350,000,000-byte ceiling. Docker Desktop can report a
smaller compressed/containerd size, so only the Linux Engine figure is used for the release
limit.
